## গল্প

ধরো তুমি প্রেমপত্র পাঠাচ্ছো ডাকে। 😄 খোলা postcard-এ লিখলে কী হবে? পথের প্রতিটা ডাক কর্মী, পিয়ন — সবাই পড়ে ফেলতে পারবে! শুধু পড়া না — চাইলে কেউ মাঝপথে লেখাটা **বদলেও** দিতে পারে: "দেখা হবে শুক্রবারে"-কে বানিয়ে দিলো "আর কোনোদিন দেখা হবে না"। সর্বনাশ!

তাহলে কী করবে? প্রথম বুদ্ধি: একটা **তালাওয়ালা বাক্স**, যার চাবি শুধু তোমাদের দুজনের কাছে। কিন্তু নতুন সমস্যা — চাবিটা ওর কাছে **পৌঁছাবে কীভাবে**? ডাকে চাবি পাঠালে তো পথেই কেউ কপি করে নেবে!

এবার আসল জাদু। ধরো প্রাপক তোমাকে পাঠালো একটা **খোলা তালা** — চাবি নিজের কাছেই রাখলো। যে কেউ এই তালা দেখতে পারে, কপি করতে পারে — লাভ নেই, কারণ তালা দিয়ে শুধু **আটকানো** যায়, খোলা যায় না। তুমি চিঠিটা বাক্সে ভরে ওই তালায় আটকে পাঠালে। পথে কেউ আর খুলতে পারবে না — খুলতে পারবে শুধু সে-ই, যার কাছে চাবি। এটাই **public key cryptography**: খোলা তালা = public key (সবাইকে দেওয়া যায়), চাবি = private key (কাউকে না)।

আর শেষ প্রশ্ন — খোলা তালাটা যে **আসল প্রাপকেরই**, ভুয়া কারো না, সেটার গ্যারান্টি কে দেবে? দরকার একজন বিশ্বস্ত নোটারি, যে সিলমোহর দিয়ে বলবে: "এই তালা সত্যিই অমুকের।" Internet-এ এই নোটারির নাম **Certificate Authority**। এই তিন বুদ্ধি মিলেই HTTPS।

## Concept

**HTTP-র সমস্যা:** আগের lesson-এর সব request/response যায় plain text-এ — postcard! একই Wi-Fi-র যে কেউ, ISP, পথের যেকোনো node তোমার password, message পড়তে পারে (eavesdropping), বদলাতে পারে (tampering), এমনকি নিজেকে golpogram.com সাজিয়ে বসে থাকতে পারে (impersonation)। **HTTPS = HTTP + TLS** — তিনটারই সমাধান। (TLS-এর পুরনো নাম SSL — নামটা এখনো মুখে মুখে চলে।)

**দুই রকম তালাচাবি:**

|        | Symmetric                      | Asymmetric                                  |
| ------ | ------------------------------ | ------------------------------------------- |
| চাবি   | একটাই — দুজনের কাছে same       | জোড়া — public (খোলা তালা) + private (চাবি) |
| গতি    | খুব দ্রুত ⚡                   | ধীর, ভারী হিসাব                             |
| ঝামেলা | চাবিটা নিরাপদে পৌঁছাবে কীভাবে? | ধীরগতি                                      |

TLS-এর চালাকি: **দুটোই ব্যবহার করো!** ধীর asymmetric দিয়ে শুধু একটা ছোট্ট জিনিস নিরাপদে আদান-প্রদান করো — একটা নতুন symmetric চাবি (session key)। তারপর বাকি পুরো আলাপ চলুক দ্রুত symmetric তালায়।

**TLS handshake — ধাপে ধাপে:**

1. **Client Hello** — browser বলে: "HTTPS চাই! আমি এই এই encryption পারি।"
2. **Server Hello + Certificate** — server বেছে নেয় পদ্ধতি, পাঠায় তার **certificate** (ভেতরে public key + CA-র সিলমোহর)।
3. **Certificate যাচাই** — browser দেখে: সিলমোহরটা কি বিশ্বস্ত CA-র? নামটা কি সত্যিই `golpogram.com`? মেয়াদ আছে? কোনোটা না মিললেই সেই ভয়ের লাল warning পেজ।
4. **Key exchange** — দুজনে মিলে (server-এর public key/দ্রুত গণিত ব্যবহার করে) একটা **session key** বানায়, যা পথের কেউ বের করতে পারে না।
5. **🔒 Encrypted!** — এখন থেকে সব HTTP traffic ওই session key-র symmetric তালায়। বাইরের কেউ দেখলে দেখবে শুধু অর্থহীন আবর্জনা।

পুরো নাটক শেষ হয় মিলিসেকেন্ডে, তোমার চোখে পড়ে শুধু ঠিকানার পাশের 🔒।

**Certificate Authority (CA) — বিশ্বাসের শিকড়:** Browser/OS-এর ভেতরে আগে থেকেই গাঁথা থাকে কিছু বিশ্বস্ত CA-র তালিকা (যেমন Let's Encrypt, DigiCert)। CA certificate দেওয়ার আগে প্রমাণ নেয় যে domain-টা সত্যিই তোমার। তাই ভুয়া কেউ `golpogram.com`-এর বৈধ certificate বানাতে পারে না — বানালে browser-ই ধরে ফেলে। বিশ্বাসটা এভাবে শিকড় থেকে পাতায় গড়ায়: OS বিশ্বাস করে CA-কে → CA সই করেছে certificate-এ → অতএব browser বিশ্বাস করে server-কে।

**HTTPS কী দেয় — তিন রক্ষা:**

- **Encryption** — কেউ পড়তে পারে না (প্রেমপত্র surf-safe! 💌)
- **Integrity** — পথে এক অক্ষর বদলালেও ধরা পড়ে
- **Authentication** — তুমি সত্যিই আসল server-এর সাথেই কথা বলছো

মনে রেখো সীমাটাও: HTTPS পথ পাহারা দেয়, **গন্তব্য না**। Server নিজেই দুষ্টু হলে বা hack হলে HTTPS বাঁচাবে না। আর কে কোন **site-এ** যাচ্ছো (domain), সেটা DNS-এর কল্যাণে অনেকে দেখেই ফেলে — ভেতরের content দেখে না।

## Real World

**Let's Encrypt-এর বিপ্লব:** ২০১৫-র আগে certificate কিনতে বছরে $৫০-৫০০ লাগতো — ছোট site-গুলো তাই HTTP-তেই পড়ে থাকতো। অলাভজনক **Let's Encrypt** এলো সম্পূর্ণ ফ্রি, automated certificate নিয়ে — আজ ওয়েবের ৯০%+ পেজ load হয় HTTPS-এ, আর Let's Encrypt একাই কোটি কোটি certificate issue করে। তোমার GolpoGram-ও deploy-এর দিন ফ্রি certificate পাবে (Vercel নিজেই ব্যবস্থা করে দেয়!)।

**Browser-দের কড়াকড়ি:** Chrome ২০১৮ থেকে সব HTTP সাইটে সরাসরি "Not Secure" লিখে দেয়, আর password field থাকলে রীতিমতো লাল সতর্কতা। ফলাফল: HTTPS এখন optional বিলাসিতা না — ভিত্তিমূল।

## মনে রাখো

- HTTP = খোলা postcard; HTTPS = HTTP + TLS — পড়া, বদলানো, সাজা — তিনটাই আটকায়।
- Public key = খোলা তালা (সবাইকে দাও), private key = চাবি (কাউকে না)।
- Asymmetric ধীর, symmetric দ্রুত — TLS তাই asymmetric দিয়ে session key বিনিময় করে, বাকি আলাপ symmetric-এ।
- Handshake: Client Hello → Server Hello + Certificate → যাচাই → key exchange → 🔒 encrypted।
- Certificate Authority (CA) হলো নোটারি — browser-এর built-in বিশ্বাসের তালিকা থেকেই সব বিশ্বাস গড়ায়।
- HTTPS দেয়: encryption + integrity + authentication; কিন্তু server নিজে খারাপ হলে রক্ষা নেই — পথ নিরাপদ, গন্তব্যের দায় আলাদা।
