## গল্প

বসুন্ধরা সিটির সিনেমা হলে গেছো তুমি। টিকেট কেটে ভেতরে ঢুকলে। Interval-এ পপকর্ন কিনতে বাইরে এলে — এখন আবার ঢুকবে কীভাবে? গেটের ভাইয়া তো তোমাকে চেনে না, আর সে শত শত মুখ মনে রাখতেও পারবে না!

হল-গুলো এটা দুইভাবে সামলায়:

**পদ্ধতি ১ — টোকেন আর খাতা:** ঢোকার সময় তোমাকে একটা নম্বর লেখা টোকেন দেওয়া হলো — "৪৭"। আর কাউন্টারের খাতায় লেখা হলো: "টোকেন ৪৭ = Gold seat, E-12, paid"। ফেরার সময় টোকেন দেখালে ভাইয়া খাতা মিলিয়ে বলে — "যান ভাই।" টোকেনে নিজে কিছুই লেখা নেই, **সব তথ্য খাতায়**।

**পদ্ধতি ২ — হাতের স্ট্যাম্প:** ঢোকার সময় হাতে বিশেষ কালির স্ট্যাম্প — যাতে লেখা "GOLD-E12", আর সেটা এমন বিশেষ সিলে দেওয়া যে নকল করা যায় না। ফেরার সময় ভাইয়া শুধু স্ট্যাম্পটা **দেখেই** ছেড়ে দেয় — কোনো খাতা খোলার দরকারই নেই। **সব তথ্য তোমার হাতেই**, আর সিলের কারিগরিই নিশ্চিত করে কেউ বদলায়নি।

মনে আছে HTTP stateless — server (গেটের ভাইয়া) কাউকে মনে রাখে না? তাহলে GolpoGram-এ একবার login করার পর প্রতি পাতায় আবার password চায় না কেন? কারণ এই দুই পদ্ধতিই! টোকেন-খাতা হলো **session**, হাতের স্ট্যাম্প হলো **JWT**। আর টোকেনটা যে পকেটে থাকে, তার নাম **cookie**।

## Concept

**Cookie — browser-এর পকেট:** Server response-এ বলতে পারে: `Set-Cookie: token=abc123`। Browser সেটা রেখে দেয় এবং **ওই site-এ প্রতিটা পরের request-এ নিজে থেকেই পাঠায়**। ব্যস — stateless HTTP-র উপরে "মনে রাখার" কৌশল পাওয়া গেলো। Cookie নিজে কোনো পদ্ধতি না — স্রেফ বহনকারী পকেট; ভেতরে session ID-ও থাকতে পারে, JWT-ও।

জরুরি cookie-flags (interview-প্রিয়!): `HttpOnly` (JavaScript ছুঁতে পারবে না — চুরি ঠেকায়), `Secure` (শুধু HTTPS-এ যাবে), `SameSite` (অন্য site থেকে আসা request-এ যাবে না — CSRF ঠেকায়)।

**Session — টোকেন আর খাতা:**

1. Login সফল → server একটা এলোমেলো **session ID** বানায় (`sess_8f3k2...`), আর নিজের খাতায় (memory/database/Redis) লেখে: `sess_8f3k2 → user 42, role: admin`
2. ID-টা cookie-তে করে browser-এ যায়
3. প্রতি request-এ cookie ফিরে আসে → server খাতা দেখে চেনে: "ও, user 42!"

সুবিধা: **নিয়ন্ত্রণ সম্পূর্ণ server-এর হাতে** — খাতা থেকে লাইনটা মুছলেই সেই user সাথে সাথে logout (চুরি যাওয়া টোকেনও অকেজো!)। অসুবিধা: **প্রতি request-এ খাতা খুলতে হয়** — মানে storage lookup। আর server ১০টা হলে? সবাইকে একই খাতা দেখতে হবে — হয় shared store (যেমন Redis) লাগবে, নয়তো ঝামেলা। খাতাটাই এখানে **state** — scaling-এর কাঁটা।

**JWT (JSON Web Token) — হাতের স্ট্যাম্প:** তিন অংশ, দুটো ডট দিয়ে জোড়া: `header.payload.signature`

- **Header:** কোন সই-পদ্ধতি (যেমন HS256)
- **Payload:** তথ্যগুলো — `{ "sub": "42", "name": "Nobel", "exp": 1718000000 }` (user কে, মেয়াদ কখন শেষ)
- **Signature:** header+payload-কে server-এর **গোপন চাবি** দিয়ে সই — এক অক্ষর বদলালেই সই আর মিলবে না

প্রতি request-এ JWT আসে (সাধারণত `Authorization: Bearer <token>` header-এ)। Server শুধু **সই মিলিয়ে দেখে** — মিললেই payload বিশ্বাসযোগ্য। **কোনো খাতাই নেই!** যেকোনো server, এমনকি নতুন চালু হওয়া server-ও শুধু গোপন চাবিটা জানলেই token যাচাই করতে পারে — stateless, scaling-বান্ধব। 🚀

⚠️ দুটো মারাত্মক ভুল ধারণা: (১) JWT-র payload **encrypted না** — শুধু base64-encoded; যে কেউ পড়তে পারে। তাই password বা গোপন কিছু কখনো payload-এ না! সই **বদলানো** ঠেকায়, **পড়া** না। (২) JWT মেয়াদের আগে **বাতিল করা যায় না** — স্ট্যাম্প তো হাত থেকে মোছা যায় না! চুরি হলে মেয়াদ (exp) পর্যন্ত সেটা বৈধ। এজন্যই access token-এর মেয়াদ রাখা হয় ছোট (মিনিট-ঘণ্টা), সাথে থাকে আলাদা **refresh token** নতুনটা নেওয়ার জন্য।

**কে কখন জেতে?**

|                    | Session 📒                      | JWT 🖐                     |
| ------------------ | ------------------------------- | -------------------------- |
| তথ্য থাকে          | Server-এর খাতায়                | Token-এর ভেতরেই            |
| প্রতি request-এ    | খাতা lookup                     | শুধু সই যাচাই              |
| সাথে সাথে বাতিল    | ✅ সহজ                          | ❌ কঠিন                    |
| বহু server-এ scale | Shared store লাগে               | এমনিই চলে                  |
| মানানসই            | নিয়ন্ত্রণ-প্রধান app (banking) | API, microservices, mobile |

আর হ্যাঁ — তোমার **GolpoGram এই মুহূর্তে JWT-ই ব্যবহার করছে**! Login-এ NestJS backend token সই করে দেয়, browser রাখে, প্রতি API call-এ `Authorization: Bearer ...` হয়ে ফেরত আসে। নিজের বানানো জিনিস দিয়েই শেখা — এর চেয়ে মজা কী! 😄

## Real World

**বড়রা কী করে:** Facebook/Amazon-এর মতো giant-রা মূল ওয়েব login-এ এখনো cookie+session ঘরানার দিকেই ঝোঁকে — কোটি user-কে **সাথে সাথে** logout/block করার ক্ষমতাটা তাদের কাছে সোনার চেয়ে দামি। অন্যদিকে Google/Auth0/Firebase-এর API জগৎ আর "Sign in with Google" (OAuth/OpenID Connect) দাঁড়িয়ে আছে JWT-জাতীয় signed token-এর উপর — কারণ হাজারো ভিন্ন server-কে খাতা না দেখেই যাচাইয়ের ক্ষমতা দিতে হয়।

**GitHub-এর token ফাঁস শিকার:** Public repo-তে ভুলে token/secret commit হয়ে যাওয়া এত সাধারণ যে GitHub এখন push-এর সময়ই secret scan করে আটকে দেয়। শিক্ষা: token = চাবি; code-এ চাবি রাখতে নেই (এজন্যই তোমার JWT_SECRET থাকে `.env`-এ, যেটা git-এ যায় না!)।

## মনে রাখো

- HTTP stateless, তাই "আমাকে মনে রাখো"-র কৌশল লাগে — বহনকারী হলো cookie (browser নিজেই প্রতি request-এ পাঠায়)।
- Session = টোকেন + server-এর খাতা: নিয়ন্ত্রণ পুরো server-এর, সাথে সাথে বাতিল করা যায়, কিন্তু প্রতি request-এ lookup আর scale-এ shared store লাগে।
- JWT = সই করা স্ট্যাম্প (`header.payload.signature`): server-এ খাতা লাগে না — stateless ও scale-বান্ধব; কিন্তু মেয়াদের আগে বাতিল করা কঠিন।
- JWT-র payload পড়া যায় (base64) — গোপন তথ্য রাখা নিষেধ; signature শুধু বদল ঠেকায়।
- তাই JWT-র মেয়াদ ছোট রাখো + refresh token ব্যবহার করো; cookie-তে রাখলে HttpOnly + Secure + SameSite।
- 401 = পরিচয় নেই/ভুল (authentication), 403 = পরিচয় আছে কিন্তু অনুমতি নেই (authorization) — গত lesson-এর সূত্র এখানে পূর্ণ হলো।
