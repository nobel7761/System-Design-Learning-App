## গল্প

আবার সেই কাচ্চির restaurant-এ ফিরি — তবে এবার অন্য চোখে। ভেবে দেখো, রান্নাঘরে কী চলছে তুমি জানো না: কয়টা চুলা, কে মশলা মাপে, ক'জন বাবুর্চি — কিছুই না। তোমার দরকারও নেই! তোমার হাতে আছে শুধু **menu** — আর সেটাই যথেষ্ট।

Menu আসলে একটা **চুক্তি**: "এই এই জিনিস চাওয়া যাবে, এভাবে চাইতে হবে, আর এমন জিনিস পাবে।" Menu-তে আছে বলেই তুমি জানো "কাচ্চি — হাফ/ফুল" অর্ডার করা যায়; menu-তে নেই বলেই জানো এখানে sushi চেয়ে লাভ নেই। রান্নাঘর কাল পুরো বদলে গেলেও — নতুন বাবুর্চি, নতুন চুলা — menu একই থাকলে তোমার অর্ডারের অভ্যাস এতটুকু বদলায় না।

Software-এর জগতে এই menu-টার নাম **API (Application Programming Interface)** — এক program আরেক program-এর সাথে কথা বলার প্রকাশ্য চুক্তি। ভেতরের জটিলতা লুকানো, বাইরে শুধু পরিষ্কার নিয়ম। আর ওয়েবের জগতে এই menu লেখার সবচেয়ে জনপ্রিয় রীতির নাম **REST**।

## Concept

**API = চুক্তি।** তোমার GolpoGram-এর frontend (Next.js) আর backend (NestJS) আলাদা দুটো program — তারা কথা বলে API দিয়ে: frontend জানে `POST /api/auth/login`-এ email-password পাঠালে token আসবে। Backend-এর ভেতরে NestJS না Django, MongoDB না Postgres — frontend-এর জানার দরকারই নেই। **চুক্তি ঠিক থাকলে দুই পাশ স্বাধীনভাবে বদলানো যায়** — এটাই API-র আসল শক্তি, আর বড় system-এ দল-ভাগ করে কাজ করার ভিত্তি।

**REST (REpresentational State Transfer)** — HTTP-কেই সুন্দরভাবে ব্যবহারের কিছু রীতি:

**নিয়ম ১ — সবকিছু resource, আর resource মানে বিশেষ্য (noun):** URL-এ থাকবে _জিনিসের_ নাম, _কাজের_ নাম না।

```
✅ /api/lessons          ❌ /api/getLessons
✅ /api/lessons/w0l7     ❌ /api/fetchLessonById?x=w0l7
✅ /api/users/42/posts   ❌ /api/getUserPostsList
```

**নিয়ম ২ — কাজ বোঝাবে HTTP method (গত lesson-এর verbs!):**

| চাওয়া                | Request                 |
| --------------------- | ----------------------- |
| সব lesson-এর তালিকা   | `GET /api/lessons`      |
| একটা নির্দিষ্ট lesson | `GET /api/lessons/w0l7` |
| নতুন post বানাও       | `POST /api/posts`       |
| Post-টা পুরো বদলাও    | `PUT /api/posts/9`      |
| শুধু caption বদলাও    | `PATCH /api/posts/9`    |
| মুছে দাও              | `DELETE /api/posts/9`   |

Noun (URL) + verb (method) — দুই মিলে পুরো বাক্য। যেকোনো নতুন developer URL দেখেই আন্দাজ করতে পারে কী ঘটবে — এটাই REST-এর সৌন্দর্য।

**নিয়ম ৩ — stateless:** প্রতিটা request স্বয়ংসম্পূর্ণ — পরিচয় (token), কী চাই, সব নিজের সাথে। আগের ৬টা lesson-এর সুতোগুলো দেখো এক জায়গায় মিলছে!

**ভাষা: JSON।** Request আর response-এর body-তে আজকের জগতে প্রায় সবসময় JSON:

```json
{ "id": "w0l7", "title": "API ও REST", "estMinutes": 30 }
```

হালকা, পড়তে সহজ, সব ভাষায় parser আছে — তাই-ই রাজা।

**বাস্তব API-র আরো তিনটা অপরিহার্য জিনিস:**

- **Versioning:** চুক্তি বদলাতে হলে পুরনো client ভাঙা চলবে না। তাই `/api/v1/posts` রেখে `/api/v2/posts` খোলা হয় — পুরনোরা v1-এ চলতে থাকে। চুক্তি ভাঙা = বিশ্বাস ভাঙা!
- **Pagination:** `GET /api/posts` — GolpoGram-এ ১ কোটি post থাকলে? সব একসাথে পাঠালে server-ও মরবে, phone-ও। তাই পাতা ভাগ: `GET /api/posts?page=3&limit=20` — "২০টা করে, ৩ নম্বর পাতা।" (Menu-ও তো এক পাতায় সব আইটেম রাখে না!)
- **ভদ্র error:** ভুল হলে শুধু `400` ছুঁড়ে না দিয়ে কারণ বলো — `{ "message": "email টা ঠিক ফরম্যাটে নেই" }`। ভালো API ব্যবহারকারীকে অন্ধকারে রাখে না।

**Idempotency — আগের সুতোটা এখানে বাঁধি:** মনে আছে, POST বারবার = বারবার নতুন জিনিস? এজন্য টাকার মতো স্পর্শকাতর API-তে client একটা **idempotency key** পাঠায় (`Idempotency-Key: abc-123`) — server দেখে এই key-র কাজ আগেই হয়েছে কিনা; হয়ে থাকলে আগের ফলটাই ফেরত দেয়, **দ্বিতীয়বার কাজ করে না**। Stripe-এর payment API এভাবেই double-charge ঠেকায়।

## Tool Spotlight

**Postman** — API-র খেলার মাঠ। GUI-তে request বানাও (method, URL, headers, body), Send চাপো, response দেখো সুন্দর করে সাজানো। Collection বানিয়ে পুরো API-র টেস্ট-খাতা রাখা যায়, environment variable-এ token রেখে বারবার ব্যবহার — backend developer-এর নিত্যসঙ্গী। হালকা বিকল্প চাইলে তো `curl` আছেই (lesson 4!), আর VS Code-এ আছে REST Client/Thunder Client extension। তোমার GolpoGram backend চালু করে Postman দিয়ে `POST http://localhost:3001/api/auth/login` মেরে দেখো — নিজের API নিজেই পরখ! 🎯

## Real World

**Stripe — API-ই যাদের পণ্য:** Stripe-এর ব্যবসাটাই একটা API: `POST /v1/charges` — টাকা কাটো। তাদের documentation এত যত্নে লেখা যে শেখার জন্যও মানুষ পড়ে! আর তাদের idempotency key-র নকশাটাই আজ industry-র পাঠ্যবই। শিক্ষা: API-র design-ই অনেক কোম্পানির আসল সম্পদ।

**Twitter API-র ট্র্যাজেডি (versioning-এর শিক্ষা):** ২০১২ আর ২০২৩-এ Twitter API-র নিয়ম/দাম হুট করে বদলে দেওয়ায় শত শত app রাতারাতি অচল হয়ে যায়। চুক্তি ভাঙলে ecosystem-এর বিশ্বাসও ভাঙে — versioning আর backward compatibility এজন্যই শুধু প্রযুক্তি না, প্রতিশ্রুতি।

## মনে রাখো

- API = menu = চুক্তি: ভেতরের জটিলতা লুকিয়ে বাইরে পরিষ্কার নিয়ম; চুক্তি ঠিক থাকলে দুই পাশ স্বাধীনভাবে বদলায়।
- REST: URL-এ resource (noun) — `/api/posts/9`; কাজ বোঝায় HTTP method (verb); প্রতিটা request stateless।
- ভাষা JSON — হালকা, সর্বজনীন।
- Versioning (`/v1`, `/v2`) — পুরনো client কখনো ভেঙো না; pagination (`?page=3&limit=20`) — সব data একসাথে কখনো না।
- Idempotency key দিয়ে POST-এর double-action ঠেকানো হয় (Stripe-স্টাইল) — payment-এ অপরিহার্য।
- ভালো error message ভালো API-র চিহ্ন: code + মানুষের ভাষায় কারণ।
