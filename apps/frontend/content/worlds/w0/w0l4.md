## গল্প

ধানমন্ডির এক restaurant-এ ঢুকলে তুমি। Waiter এলো, তুমি menu দেখে বললে — "ভাই, একটা কাচ্চি, হাফ।" Waiter order টা রান্নাঘরে দিলো, কিছুক্ষণ পর হয় গরম কাচ্চি এলো, নয়তো খবর এলো — "ভাই, কাচ্চি শেষ।" 😢

লক্ষ্য করো এই কথোপকথনের **নিয়মগুলো**: তুমি চাও → সে জবাব দেয়। জবাবটা হয় সফল ("এই নিন"), নয় ব্যর্থ ("নেই")। আর প্রতিবার অর্ডার নতুন করে দিতে হয় — গতকাল কাচ্চি খেয়েছিলে বলে আজ waiter এমনি এমনি কাচ্চি আনবে না; আজকের waiter হয়তো নতুন, সে তোমাকে চেনেই না!

Browser আর server-ও ঠিক এই restaurant-এর নিয়মে কথা বলে। সেই নিয়মের নাম **HTTP (HyperText Transfer Protocol)** — request যায়, response আসে, আর প্রতিটা লেনদেন আগেরটার থেকে স্বাধীন। আগের lesson-এর TCP যদি হয় টেলিফোন লাইন, HTTP হলো সেই লাইনে বলা **ভাষা**।

## Concept

**HTTP request-এর গঠন — অর্ডারের slip:**

```
GET /api/lessons/w0l4 HTTP/1.1        ← method + path + version
Host: golpogram.com                    ← কোন দোকানে
Authorization: Bearer eyJhbG...        ← "আমি member!" (পরের lesson-এ)
Content-Type: application/json         ← headers (বাড়তি তথ্য)

{ "..." }                              ← body (কিছু পাঠালে)
```

**Methods — কী ধরনের কাজ চাও:**

| Method     | মানে            | Restaurant-এ                        | একটা গুণ                    |
| ---------- | --------------- | ----------------------------------- | --------------------------- |
| **GET**    | data দাও        | "Menu-টা দেখান"                     | কিছু বদলায় না (safe)       |
| **POST**   | নতুন কিছু বানাও | "একটা কাচ্চির অর্ডার নিন"           | বারবার দিলে বারবার অর্ডার!  |
| **PUT**    | পুরোটা বদলে দাও | "অর্ডার বদলে পুরোটা বিরিয়ানি করুন" | একই request বারবার = একই ফল |
| **PATCH**  | একটু খানি বদলাও | "ঝাল কমিয়ে দিন শুধু"               | আংশিক পরিবর্তন              |
| **DELETE** | মুছে ফেলো       | "অর্ডারটা cancel করুন"              | একই ফল বারবার               |

ওই "বারবার দিলে একই ফল" গুণটার নাম **idempotency** — GET/PUT/DELETE idempotent, POST নয়। মনে রাখো: দু'বার submit চাপলে POST-এ **দুটো** কাচ্চি চলে আসে! API design-এ এটা বিরাট ব্যাপার — World 3-এ payment-এর গল্পে ফিরবে।

**Status codes — waiter-এর জবাব:** প্রথম digit-টাই গল্প বলে দেয়:

- **2xx — সফল!** `200 OK` (এই নিন), `201 Created` (নতুন বানালাম), `204 No Content` (করলাম, দেখানোর কিছু নেই)
- **3xx — অন্যদিকে যাও।** `301 Moved Permanently` (দোকান স্থায়ীভাবে ওই পাশে), `302 Found` (আজকের মতো ওদিকে), `304 Not Modified` (যা আছে তোমার কাছে, তা-ই তাজা — cache!)
- **4xx — ভুল তোমার।** `400 Bad Request` (অর্ডারটাই বুঝিনি), `401 Unauthorized` (আগে পরিচয় দিন), `403 Forbidden` (চিনলাম, তবু এটা আপনার জন্য না), `404 Not Found` (এই item নেই), `429 Too Many Requests` (ভাই, একটু থামেন!)
- **5xx — ভুল আমাদের।** `500 Internal Server Error` (রান্নাঘরে আগুন 🔥), `502 Bad Gateway` (মাঝের লোকটা ঝামেলা করছে), `503 Service Unavailable` (আজ দোকান overload)

Debugging-এর সোনার নিয়ম: **4xx দেখলে আগে নিজের request চেক করো, 5xx দেখলে server-এর log।**

**Statelessness — waiter-এর স্মৃতি নেই:** HTTP-র প্রতিটা request সম্পূর্ণ স্বাধীন — server আগের request-এর কিছু মনে রাখে না। এটা দুর্বলতা না, বরং **superpower**: যেকোনো request যেকোনো server-এ পাঠানো যায়! GolpoGram-এ ১০টা server থাকলে তোমার request #1 server-৩-এ আর request #2 server-৭-এ গেলেও কিছু যায় আসে না — কেউ কিছু "মনে রাখার" দায়িত্বে নেই। Scaling এত সহজ হওয়ার পেছনে এই statelessness-ই নায়ক। (তাহলে login মনে থাকে কীভাবে? দারুণ প্রশ্ন — উত্তর lesson 6-এ!)

**HTTP-র বিবর্তন (এক ঝলক):** HTTP/1.1 — এক connection-এ এক সময়ে এক request (লাইনে দাঁড়ানো)। HTTP/2 — এক connection-এ অনেক request পাশাপাশি (multiplexing)। HTTP/3 — গত lesson-এর সেই QUIC/UDP-র উপরে, আরো দ্রুত শুরু। গভীরে যাবো পরে; এখন শুধু জেনে রাখো — ভাষা এক, বাহনটা ক্রমেই দ্রুততর হয়েছে।

## Tool Spotlight

`curl` — terminal থেকে HTTP request পাঠানোর সুইস আর্মি নাইফ:

```bash
curl -i https://api.github.com/users/octocat
```

`-i` flag-এ response-এর status line আর headers-ও দেখবে — `HTTP/2 200`, `content-type: application/json`… পুরো অর্ডার-জবাব নিজের চোখে! POST পাঠাতে:

```bash
curl -X POST https://localhost:3001/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"...","password":"..."}'
```

চেনা লাগছে? 😉 এটা তোমার GolpoGram-এরই login API!

## Real World

**404-এর জন্ম:** নামটা এসেছে "পাওয়া যায়নি" অর্থের status code থেকে, আর আজ এটা pop culture-এর অংশ — GitHub-এর বিখ্যাত 404 পেজে দেখায় Star Wars-এর দৃশ্য। প্রতিটা বড় সাইটের সুন্দর custom 404 পেজ থাকে, কারণ ভুল পথে আসা user-কেও ভালো অভিজ্ঞতা দিতে হয়।

**429 আর rate limiting:** Twitter/X-এর API দিনে নির্দিষ্ট সংখ্যক request-এর বেশি করলেই পাঠায় `429 Too Many Requests` — সাথে `Retry-After` header: "এত সেকেন্ড পরে এসো।" এক client যেন সবার সেবা খেয়ে না ফেলে — এই রক্ষাকবচের পুরো গল্প World 4-এ।

## মনে রাখো

- HTTP = request → response — restaurant-এর অর্ডারের নিয়ম; TCP লাইন হলে HTTP তাতে বলা ভাষা।
- Methods: GET (পড়ো), POST (বানাও), PUT (পুরো বদলাও), PATCH (একটু বদলাও), DELETE (মুছো)।
- Idempotency: GET/PUT/DELETE বারবার = একই ফল; POST বারবার = বারবার নতুন জিনিস — সাবধান!
- Status codes: 2xx সফল, 3xx redirect, 4xx তোমার ভুল, 5xx server-এর ভুল। চেনা মুখ: 200, 201, 301, 304, 401, 403, 404, 429, 500, 503।
- HTTP stateless — server কিছু মনে রাখে না; এই গুণেই request যেকোনো server-এ পাঠানো যায় = সহজ scaling।
- Headers = মেটাডেটা (কে, কী format, কত বড়); body = আসল মাল।
