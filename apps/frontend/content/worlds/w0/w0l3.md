## গল্প

দুটো চিঠি পাঠানোর দৃশ্য কল্পনা করো।

**দৃশ্য ১:** তুমি জমির দলিল পাঠাচ্ছো রংপুরে। ডাকঘরে গিয়ে **registered post** করলে — প্রাপক সই করে বুঝে নেবেন, receipt আসবে, হারালে খোঁজ পড়বে, আর দশ পাতার দলিল ঠিক ক্রমে পৌঁছেছে কিনা নিশ্চিত করা হবে। একটু ধীর, একটু খরচ বেশি — কিন্তু **নিশ্চয়তা ১০০%**।

**দৃশ্য ২:** পাড়ার মেলার মাইকওয়ালা ঘোষণা দিচ্ছে: "আজ সন্ধ্যায় যাত্রাপালা!" সে কি প্রতিটা মানুষের কাছে গিয়ে জিজ্ঞেস করে — "ভাই, শুনেছেন তো?" না! সে শুধু বলে যায়। দু-একজন না শুনলে কিছু যায় আসে না — ঘোষণা **দ্রুত আর হালকা** হওয়াটাই আসল।

Internet-এও ঠিক এই দুটো স্টাইলে data পাঠানো যায়। Registered post হলো **TCP** — ধীর কিন্তু নির্ভরযোগ্য। মাইকের ঘোষণা হলো **UDP** — ঝড়ের গতি, কিন্তু কেউ মিস করলে করলো। কোনটা কখন — এটাই আজকের আসল শিক্ষা।

## Concept

**আগে IP-টা ঝালাই করি।** IP (Internet Protocol) হলো ঠিকানা ব্যবস্থা — প্রতিটা packet-এ source আর destination IP থাকে, router-রা সেটা দেখে পথ দেখায়। কিন্তু IP নিজে **কোনো promise করে না**: packet হারাতে পারে, এলোমেলো ক্রমে পৌঁছাতে পারে, duplicate-ও হতে পারে। IP শুধু "best effort"-এ পৌঁছে দেওয়ার চেষ্টা করে। এই অগোছালো ভিতের উপর শৃঙ্খলা আনে TCP বা UDP।

**Port — ঠিকানার flat number:** IP address-এ computer পর্যন্ত পৌঁছানো যায়, কিন্তু সেই computer-এ তো অনেক program চলছে! **Port number** বলে দেয় কোন program-এর কাছে data যাবে — যেমন একই building-এর ভিন্ন flat। চেনা port: ওয়েবের HTTPS = `443`, HTTP = `80`, আর তোমার GolpoGram backend dev server = `3001`। ঠিকানা লেখা হয় `103.108.140.22:443` — IP আর port একসাথে।

**TCP (Transmission Control Protocol) — registered post:**

TCP কথা শুরু করার আগে একটা ভদ্রতার পর্ব সারে, নাম **3-way handshake**:

```
Client:  "SYN"      → "হ্যালো, কথা বলতে চাই!"
Server:  "SYN-ACK"  → "শুনছি, তুমিও কি শুনছো?"
Client:  "ACK"      → "হ্যাঁ! চলো শুরু করি।"
```

Connection তৈরি হওয়ার পর TCP দেয় তিনটা বড় promise:

1. **Delivery guarantee** — প্রতিটা packet পৌঁছানোর confirmation (ACK) আসে; না এলে আবার পাঠানো হয় (retransmission)।
2. **Order guarantee** — sequence number মিলিয়ে data সঠিক ক্রমে সাজিয়ে দেওয়া হয়।
3. **Congestion control** — রাস্তা জ্যাম দেখলে TCP নিজে থেকেই গতি কমায়, ফাঁকা পেলে বাড়ায়।

দাম? প্রতিটা নিশ্চয়তার জন্য সময় আর bandwidth খরচ হয় — handshake-এ এক round-trip, ACK-এ বাড়তি traffic।

**UDP (User Datagram Protocol) — মাইকের ঘোষণা:**

UDP-তে handshake নেই, ACK নেই, retry নেই, order নেই। Packet ছুঁড়ে দাও — পৌঁছালে পৌঁছালো। শুনতে দায়িত্বজ্ঞানহীন লাগছে? কিন্তু ভাবো video call-এর কথা: একটা frame হারিয়ে গেলে সেটা **ফেরত চাওয়ার মানেই নেই** — ততক্ষণে কথা আরো তিন সেকেন্ড এগিয়ে গেছে! পুরনো frame retry করলে বরং সব জমে যাবে। এখানে গতিই রাজা, আর UDP-ই সঠিক পছন্দ।

|            | TCP 📜                          | UDP 📢                               |
| ---------- | ------------------------------- | ------------------------------------ |
| Connection | Handshake লাগে                  | লাগে না                              |
| Delivery   | Guaranteed (retry হয়)          | Best effort                          |
| Order      | ঠিক ক্রমে                       | যেমন পৌঁছায়                         |
| গতি        | তুলনায় ধীর                     | দ্রুত, হালকা                         |
| ব্যবহার    | Web, API, email, file, **টাকা** | Video call, gaming, live stream, DNS |

**মনে রাখার সূত্র:** ভুল হলে ক্ষতি বড় (টাকা, document, message) → TCP। পুরনো data মূল্যহীন, গতিই জীবন (live ভিডিও, game-এর position) → UDP।

মজার টুইস্ট: গত lesson-এর DNS lookup সাধারণত **UDP** দিয়ে হয় (port 53) — query ছোট, হারালে আবার জিজ্ঞেস করলেই হয়; প্রতিবার handshake করা পোষায় না।

## Real World

**WhatsApp call vs message:** একই app, দুই protocol দর্শন! Message যায় reliable পথে — একটা অক্ষরও হারানো চলবে না। কিন্তু voice/video call চলে UDP-ভিত্তিক RTP stream-এ — মাঝে দু-একটা packet হারালে সামান্য খসখস শোনা যায়, তবু call থামে না।

**Google-এর QUIC — সেরা দুই জগতের মিশেল:** Google ভাবলো — TCP-র reliability চাই, কিন্তু তার handshake-এর ধীরগতি চাই না। সমাধান: UDP-র উপরে নিজেরাই reliability-র স্তর বানালো — নাম **QUIC**, যা এখন **HTTP/3**-এর ভিত। YouTube আর Google Search-এর বড় অংশ আজ এটাতেই চলে। শিক্ষাটা সুন্দর: protocol-গুলো ইট-পাথরের মতো — দরকারে নতুন ইমারত গড়া যায়।

## মনে রাখো

- IP ঠিকানা দেয় কিন্তু promise দেয় না — packet হারাতে, এলোমেলো হতে পারে; শৃঙ্খলা আনে TCP/UDP।
- Port = flat number: IP মেশিন চেনায়, port চেনায় program (HTTPS=443, HTTP=80, DNS=53)।
- TCP = registered post: 3-way handshake (SYN → SYN-ACK → ACK), delivery + order guarantee, congestion control — নির্ভরযোগ্য কিন্তু ভারী।
- UDP = মাইকের ঘোষণা: no handshake, no retry — হালকা ও দ্রুত, ক্ষতি-সওয়া কাজের জন্য।
- সূত্র: ভুল হলে বিপদ → TCP; দেরি হলে বিপদ → UDP।
- DNS চলে UDP-তে, আর HTTP/3 (QUIC) হলো UDP-র উপরে বানানো reliable protocol — design সবসময় বিবর্তিত হয়।
