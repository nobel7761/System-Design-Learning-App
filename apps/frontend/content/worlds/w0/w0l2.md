## গল্প

ধরো তোমার বন্ধু রাফি নতুন ফোন কিনেছে, কিন্তু তার নম্বরটা তোমার মুখস্থ নেই। তুমি কী করো? Contacts app খুলে "Rafi" লিখে search দাও — app নম্বরটা বের করে দেয়, আর কল চলে যায়। তুমি কখনোই `+8801712345678` মুখস্থ করার চেষ্টা করো না; তুমি মনে রাখো শুধু **নাম**।

আগের lesson-এ শিখেছো — Internet-এ প্রতিটা computer-এর ঠিকানা হলো IP address, যেমন `142.250.193.78`। এবার ভাবো, YouTube দেখতে হলে যদি এই নম্বর মুখস্থ রাখতে হতো, Facebook-এর জন্য আরেকটা, GolpoGram-এর জন্য আরেকটা — জীবনটা কেমন হতো? 😵

এই সমস্যার সমাধানই হলো **DNS (Domain Name System)** — পুরো Internet-এর জন্য একটা বিশাল, সবার ভাগাভাগি করা phonebook। তুমি মনে রাখবে `youtube.com` (নাম), আর DNS তোমাকে বের করে দেবে `142.250.193.78` (নম্বর)। প্রতিবার তুমি browser-এ কোনো ঠিকানা লেখো, পেছনে এই phonebook-এ একটা খোঁজাখুঁজি হয় — এক সেকেন্ডের কয়েক ভাগের মধ্যে!

## Concept

**DNS lookup আসলে কীভাবে হয়?** মজার ব্যাপার হলো, পুরো পৃথিবীর সব domain একটা খাতায় লেখা নেই। খোঁজাটা হয় ধাপে ধাপে — অনেকটা এভাবে: তুমি একটা অচেনা গ্রামের ঠিকানা খুঁজছো; প্রথমে জিজ্ঞেস করলে জেলা অফিসে, তারা পাঠালো উপজেলায়, তারা পাঠালো গ্রামের চেয়ারম্যানের কাছে — চেয়ারম্যানই শেষে বাড়িটা চিনিয়ে দিলেন।

`blog.golpogram.com` খুঁজতে গেলে ধাপগুলো হয়:

1. **Browser cache** — "এই নামটা কি আমি আগেই খুঁজেছি?" থাকলে এখানেই শেষ।
2. **Resolver (তোমার ISP বা 8.8.8.8)** — তোমার হয়ে খোঁজার দায়িত্ব নেয়। একে বলে recursive resolver।
3. **Root server** — পৃথিবীতে ১৩ সেট আছে। বলে: "`.com`? ওটা TLD server-এর কাজ, ওদিকে যাও।"
4. **TLD server** (`.com`-এর দায়িত্বে) — বলে: "`golpogram.com`? ওর authoritative server হলো অমুক।"
5. **Authoritative server** — আসল উত্তরটা এর কাছেই: "`blog.golpogram.com` = `103.108.140.22`"

Resolver উত্তরটা তোমার browser-কে দেয়, আর **cache করে রাখে** — পরের জন যখন একই নাম খুঁজবে, root পর্যন্ত আর যেতে হবে না।

**TTL (Time To Live):** প্রতিটা DNS উত্তরের সাথে একটা মেয়াদ থাকে — যেমন ৩০০ সেকেন্ড। মানে: "এই উত্তরটা ৫ মিনিট পর্যন্ত বিশ্বাস করতে পারো, তারপর আবার জিজ্ঞেস কোরো।" TTL ছোট হলে পরিবর্তন দ্রুত ছড়ায় কিন্তু DNS server-এ চাপ বাড়ে; বড় হলে উল্টোটা। দেখলে? প্রথম দিনেই system design-এর প্রিয় শব্দটা পেয়ে গেলে — **trade-off**!

**Record types — phonebook-এর ভিন্ন ভিন্ন এন্ট্রি:**

| Record    | কাজ                                  | উদাহরণ                               |
| --------- | ------------------------------------ | ------------------------------------ |
| **A**     | নাম → IPv4 address                   | `golpogram.com → 103.108.140.22`     |
| **AAAA**  | নাম → IPv6 address                   | `golpogram.com → 2406:da18::1`       |
| **CNAME** | নাম → আরেকটা নাম (ডাকনাম)            | `www.golpogram.com → golpogram.com`  |
| **MX**    | এই domain-এর email কোথায় যাবে       | `golpogram.com → mail.google.com`    |
| **NS**    | এই domain-এর authoritative server কে | `golpogram.com → ns1.cloudflare.com` |

**DNS কেন system design-এ এত গুরুত্বপূর্ণ?** কারণ এটা শুধু phonebook না — একটা **নিয়ন্ত্রণ কক্ষ**। ভবিষ্যতে শিখবে: একই নামের পেছনে অনেকগুলো IP রেখে load ভাগ করা যায় (DNS load balancing), user-কে তার কাছের server-এ পাঠানো যায় (geo-routing), আর server বদলালে শুধু DNS record বদলালেই চলে — user-রা টেরও পায় না।

## Tool Spotlight

তোমার computer-এ এক্ষুনি এটা দেখতে পারো! Terminal খুলে লেখো:

```
dig google.com
```

উত্তরে `ANSWER SECTION`-এ দেখবে A record আর তার TTL:

```
google.com.    138    IN    A    142.250.193.78
```

ওই `138` হলো TTL — আর কত সেকেন্ড এই উত্তরটা cache-এ তাজা থাকবে। দুইবার পরপর চালাও — দেখবে সংখ্যাটা কমছে! `dig +trace google.com` চালালে root → TLD → authoritative পুরো যাত্রাটা নিজের চোখে দেখতে পাবে। (Windows-এ `nslookup google.com`।)

## Real World

**Facebook-এর মহাবিপর্যয় (অক্টোবর ২০২১):** এক configuration ভুলে Facebook নিজেদের DNS route-গুলোই Internet থেকে মুছে ফেলেছিল। ফলাফল? `facebook.com`, WhatsApp, Instagram — ৬ ঘণ্টা পুরো গায়েব। মজার (এবং ভয়ের) ব্যাপার: তাদের office-এর দরজার security system-ও ওই DNS-এর উপর নির্ভর করতো — engineer-রা server room-এ ঢুকতেই পারছিল না! শিক্ষা: DNS ভাঙলে সব ভাঙে।

**Cloudflare-এর 1.1.1.1:** Cloudflare একটা public resolver চালায় যেটা গড়ে ~১৪ ms-এ উত্তর দেয়। পৃথিবীজুড়ে শত শত শহরে একই IP (1.1.1.1) announce করা থাকে — তুমি যেখানেই থাকো, কাছের server-টাই উত্তর দেয়। এই কৌশলের নাম anycast — পরে বিস্তারিত শিখবে।

## মনে রাখো

- DNS = Internet-এর phonebook: মানুষ মনে রাখে নাম (`youtube.com`), machine চলে IP address-এ।
- Lookup-এর সিঁড়ি: browser cache → resolver → root → TLD → authoritative server।
- Resolver উত্তর **cache** করে, আর **TTL** বলে দেয় উত্তরটা কতক্ষণ তাজা।
- দরকারি record: A (নাম→IPv4), AAAA (IPv6), CNAME (ডাকনাম), MX (email), NS (কে authoritative)।
- TTL ছোট = দ্রুত পরিবর্তন কিন্তু বেশি query; TTL বড় = কম চাপ কিন্তু ধীর পরিবর্তন — classic trade-off।
- DNS ভাঙলে নাম ধরে কাউকে খুঁজে পাওয়া যায় না — তাই DNS-ই অনেক বড় outage-এর কেন্দ্রে।
